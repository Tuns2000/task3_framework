# Практическая работа 3 Шаблоны проектов

Внутри архива два шаблона, один на C#, второй на Node.js.

Оба шаблона показывают работу с настройками из трёх источников, проверку корректности настроек до запуска, базовую защиту для запросов из браузера, ограничение частоты запросов и добавление защитных заголовков.

---

## Отчёт по выполнению работы (Node.js)

### 1. Инструкция запуска для обоих режимов

Приложение поддерживает чтение конфигурации из 3 источников в следующем приоритете: **Аргументы командной строки > Переменные окружения > Файл `appsettings.json`**. 
Для переключения режимов без изменения кода нужно передать соответствующий параметр `mode` (либо через переменную окружения `APP_MODE`, либо флагом `--mode=...`).

**Учебный режим (Development)**
В учебном режиме клиенту возвращаются подробные сообщения об ошибках.
Запуск через командную строку:
```bash
node src/server.js --mode=учебный
```
Также можно использовать переменные окружения (в PowerShell):
```powershell
$env:APP_MODE="учебный"; node src/server.js
```

**Боевой режим (Production)**
В боевом режиме приложение строго ограничивает выдаваемую информацию (возвращает скупое «Ошибка обработки запроса»).
Запуск через командную строку:
```bash
node src/server.js --mode=боевой
```
Также можно переопределить порты и лимиты прямо при запуске, например так:
```bash
node src/server.js --mode=боевой --port=8080 --trustedOrigins=https://my-frontend.com --readPerMinute=100
```

### 2. Сценарии ручных проверок (испытания)

Помимо встроенных автоматических тестов (`npm test`), подтверждающих логику настройки приоритетов и блокировок, вы можете провести ручные проверки:

**1. Проверка: некорректная конфигурация останавливает запуск**
Запускаем службу с недопустимым портом (`0`) или несуществующим режимом работы:
```bash
node src/server.js --mode=тест --port=0
```
*Результат:* Приложение аварийно завершается до открытия порта, выводя в консоль понятную причину:
```text
Запуск остановлен из за некорректных настроек
- Режим работы задан неверно, допустимы учебный и боевой
- Порт задан неверно, значение должно быть целым числом от 1 до 65535
```

**2. Проверка: блокировка запросов с недоверенного источника (CORS)**
Отправляем запрос, подменяя заголовок `Origin` неизвестным доменом:
```bash
curl -i -H "Origin: http://hacker.network" http://localhost:3000/api/items
```
*Результат:* Сервер ответит, но в заголовках ответа **не будет** `Access-Control-Allow-Origin`, из-за чего браузер заблокирует чтение ответа на стороне клиента.

**3. Проверка: реакция на превышение лимитов (Rate Limit)**
Лимит на создание записей (`POST` на `/api/items`) по умолчанию равен 20 в минуту. Можно временно переопределить его на 1 для проверки:
```bash
node src/server.js --writePerMinute=1
```
Далее отправляем два подряд POST запроса в другом терминале:
```bash
# Первый запрос - успешный (201 Created)
curl -X POST http://localhost:3000/api/items -d '{"name":"test", "price":10}'

# Второй запрос сразу же - заблокирован (429 Too Many Requests)
curl -i -X POST http://localhost:3000/api/items -d '{"name":"test", "price":10}'
```
*Результат:* Второй запрос получает статус `429 Too Many Requests` и текст ответа `Слишком много запросов`.

**4. Проверка: влияние режима на детализацию ошибок**
Запрашиваем с неправильным телом (без имени) в учебном режиме:
```bash
node src/server.js --mode=учебный
curl -i -X POST http://localhost:3000/api/items -d '{"price":10}'
```
*Результат ответа:* `HTTP/1.1 400 Bad Request` и текст: `Поле name не должно быть пустым`

Теперь проверяем то же самое в боевом режиме:
```bash
node src/server.js --mode=боевой
curl -i -X POST http://localhost:3000/api/items -d '{"price":10}'
```
*Результат ответа:* `HTTP/1.1 400 Bad Request` и скрытый безопасный текст: `Ошибка обработки запроса`. Таким образом подтверждается, что параметры реально отличаются и влияют на поведение.

### 3. Пояснительная записка (Реализация требований)

**1. Чтение настроек из трёх источников с явным порядком приоритета**  
В приложении реализовано чтение конфигурации из: файла `appsettings.json`, переменных окружения (Environment) и аргументов командной строки.  
Слияние конфигураций происходит в функции `buildConfig` в файле `src/config.js`. Приоритет применения (от низшего к высшему) строго задан: файлы -> переменные окружения -> аргументы запуска. Сначала загружаются данные из файла, а затем сверху накладываются (перезаписывая предыдущие) значения из других источников.

**2. Ранняя проверка корректности настроек (Fail-Fast) и усложнение на логику URL**  
В файле `src/config.js` реализована функция `validateConfig`, которая проверяет типы, лимиты, а также валидирует список доверенных источников через класс `URL` (проверка спецификации http/https, защита от опечаток).  
Вызов проверки происходит синхронно до того, как открывается сокет. Приложение не запустится при наличии ошибок, выводя понятную причину.

**3. Ограничение запросов из браузера (CORS)**  
Блокировка неавторизованных доменов реализована в файле `src/security.js`. При каждом входящем запросе проверяется заголовок `Origin`, и только если он есть в доверенном списке (`trustedOrigins`), сервер вернёт разрешающий заголовок `Access-Control-Allow-Origin`.

**4. Ограничение частоты запросов (Rate Limit) с разными лимитами**  
Для защиты от DDoS-атак и спама реализован Rate Limiter с хранением состояния в памяти. Сделано усложнение со звездочкой: разделены лимиты для маршрутов на чтение (GET) и на создание (POST), так как операции записи обычно ресурсоёмкие.  
В `src/server.js` этот лимитер вызывается до любой бизнес-логики.

**5. Добавление защитных заголовков (Security Headers)**  
В файле `src/security.js` прописаны заголовки для запрета встраивания службы в iframe (защита от Clickjacking), запрета кеширования на уровне браузера и сниффинга MIME-типов.

**6. Два режима работы (Учебный и Боевой)**  
Смена режима происходит простым изменением настройки `--mode`. Переключение меняет поведение приложения, в частности — детализацию сообщений об ошибках клиенту внутри блока catch. Никакого переписывания кода не требуется.

### Выводы (Ответы на вопросы из задания)

**1. Какие настройки вы считаете критичными и почему?**  
Самыми критичными являются списки `trustedOrigins` (CORS) и `rateLimits`.
* Если `trustedOrigins` будет иметь ошибку (например `*`), API станет уязвим к атакам с чужих доменов (CSRF/межсайтовая кража данных).
* Отсутствие или бесконечные лимиты в `rateLimits` завалят службу (DDoS-атака или скрипты-парсеры могут исчерпать ресурсы ОЗУ/БД сервера).

**2. Как обеспечено раннее обнаружение ошибок конфигурации и почему это снижает риски?**  
Применена концепция **Fail-Fast**: все проверки конфигураций (правильность портов, валидность URL-адресов) выполняются синхронно *ДО* того, как будет активирован HTTP-сервер (`server.listen()`). При любой опечатке в настройках приложение аварийно завершится (Exit Code 1).  
Это снижает риски для бизнеса: ошибка будет обнаружена на этапе развертывания (в CI/CD пайплайне), не позволив выложить в открытый доступ службу с «дырявой» конфигурацией. Использовать неверно настроенный API потенциальным злоумышленникам будет физически невозможно — сервер просто не запустится.
